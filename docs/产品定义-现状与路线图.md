# RuleDoctor 产品定义：今天是什么、不是什么、往哪走

> 读者：产品负责人 / 第一次接触的用户  
> 代码版本：v0.1.x（桌面工作区 `rule-doctor/`）

---

## 10 秒版（给用户看的 Slogan）

**今天（v0.1）：**  
「Claude Code 干完活之后，用会话日志 + 项目规则文件做一次**规则体检**——告诉你哪些规则根本没进上下文、哪些在代码/命令里已经违规。」

**不是：** 实时盯屏、自动改 Agent 行为、统一管理所有平台的 Skills/Scale。  
**目标（v1）：** 「给 Claude Code / Cursor 装一个**规则监督员**」——任务进行中就能看到红绿灯，出事时能一键纠偏。

---

## 一、「规则」到底指什么？

### 1.1 在 RuleDoctor 里的操作定义（可执行、可复现）

**规则 = 从「明文规则文件」里解析出来的一条条可检查陈述。**

每条规则有：

| 字段 | 含义 |
|------|------|
| `id` | `R1`…`Rn`（按解析顺序） |
| `text` | 规则正文（列表项或句子） |
| `section` | 所属 Markdown 标题（若有） |
| `source` | 来自哪个文件的绝对路径 |

解析方式：读文件 → 优先 Markdown 列表项 → 否则按句子切分（见 `src/rules.ts`）。**不是** LLM 归纳的「隐含约束」。

### 1.2 v0.1 **会**自动当作规则来源的文件（项目根目录）

按发现顺序扫描，**全部合并**（不是二选一）：

- `CLAUDE.md`
- `AGENTS.md`
- `.cursorrules`
- `GEMINI.md`
- `.windsurfrules`
- `copilot-instructions.md`

可用 `--rules a.md,b.md` 手动指定。

### 1.3 v0.1 **不会**当作规则来源的（必须写清楚，避免误解）

| 来源 | 现状 | 说明 |
|------|------|------|
| Claude **Skills** / Scale 工作流里的步骤 | ❌ 不读 | 除非你把同样文字写进 `CLAUDE.md` 等 |
| `.cursor/rules/*.mdc`、`.claude/rules/` | ❌ 不扫 | 待 v0.2 适配 |
| `~/.claude/settings.json`、hooks、权限配置 | ❌ | 属于「机制」不是「条文」 |
| 用户全局 Cursor Rules、Claude 账号级偏好 | ❌ | 不在项目目录 |
| **System prompt**、厂商内置安全策略 | ❌ 不解析 | 仅当**原文出现在会话 jsonl** 时，可能参与「读到率」关键词匹配 |
| 会话里用户随口说的「帮我注意 xxx」 | ⚠️ 部分 | 会进 transcript，但**不会**自动变成一条「规则」除非也写在规则文件里 |
| 代码注释、README 里的规范 | ❌ 默认不算 | 除非你把它写进规则文件 |

### 1.4 优先级与冲突（诚实回答）

**v0.1 没有优先级引擎。** 多文件同时存在时：

- 所有文件解析结果 **拼成一张列表**；
- 仅对 **完全相同的规则文本** 去重；
- **冲突**（如 A 文件说用 Tab、B 说用空格）→ **两条都保留**，不会自动裁决。

**规划（v0.3+）：** 引入 `rules manifest`：来源、优先级、覆盖关系、生效范围（路径/glob/工具）。

## 二、规则从哪来？用户怎么知道在检查什么？

### 2.1 今天怎么获得规则

- **默认：自动**扫描项目根上述文件名（存在就读）。
- **可选：** `--rules` 手动列表。
- **遵守检查** 不自动从自然语言生成，需要 **`.ruledoctor.json`** 里为关键规则配置 checker（正则 / 禁止命令）。

### 2.2 「规则面板」——今天就能用的命令

```bash
ruledoctor inventory --cwd .
```

输出：

- 发现了哪些规则文件、每条 `R#` 来自哪个文件；
- 哪些规则配置了 checker、哪些只有「读到率」没有「遵守率」；
- 明确列出 **尚未覆盖的规则来源类型**（mdc、Skills 等）。

（JSON：`ruledoctor inventory --format json`）

### 2.3 还缺什么

- 图形化规则面板（Web/TUI）→ 路线图 v0.2；
- 从规则正文 **半自动推荐 checker** → v0.3。

## 三、怎么判断「没遵守」？依据是什么？会不会误判？

### 3.1 两条独立信号（不用 LLM Judge）

| 信号 | 输入 | 算法 | 证明的是什么 |
|------|------|------|----------------|
| **读到率** | Claude Code 会话 `*.jsonl` | 把规则切成关键词/二元组，看是否出现在 flatten 后的 transcript | 规则**文本是否曾进入**（或出现在）会话语料 |
| **遵守率** | 工作区文件 + 同一份 transcript | 用户配置的 `forbid-regex` / `require-regex` / `forbid-command` | **可机械验证**的违规（如浮点、force push） |

**不用：** 对答质量做 LLM 打分、不看 git diff 专用逻辑（但改动的文件会进 regex 扫描范围）。

### 3.2 证据链（v0.1 有什么、缺什么）

| 违规类型 | 已有证据 | 缺口 |
|----------|----------|------|
| 遵守失败（regex） | 规则 id、checker 类型、**首个命中文件 + 片段预览** | 无行号（可加） |
| 遵守失败（命令） | 规则 id、禁止子串、说明「在会话中出现」 | 未标 jsonl **行号/时间戳**（可加） |
| 读到率低 | 匹配 token 数 / 总数 | 未展示「哪条 jsonl 出现过哪些 token」（可加） |
| 「看到了但无视」 | 读到率高 + 遵守失败（如 R6 force push） | 需用户配置 checker 才能判遵守 |

### 3.3 误判风险（必须对用户透明）

| 场景 | 风险 |
|------|------|
| 读到率 | 规则里常见词在别处置顶 → **虚高**；规则很短 → **不稳定** |
| 读到率 | 规则从未加载但 transcript 里碰巧出现同词 → **虚高**（少见） |
| 遵守率 | 正则过宽 → 误报；过窄 → 漏报 |
| 语义类规则（「代码要优雅」） | **v0.1 无法自动判遵守**，只能人工配 checker 或未来 probe |

**规划：** `ruledoctor explain R3` 输出命中 jsonl 行号 + 文件行号 + 可复制证据块。

## 四、产品形态：通俗、即插即用

### 4.1 今天（v0.1）的真实形态

| 维度 | 事实 |
|------|------|
| 形态 | **CLI**，事后跑一次 |
| 安装 | `npm i` / clone + build；`npx` 待 publish |
| 接入 | 不嵌入 Claude Code 进程；读磁盘上的 jsonl |
| 实时性 | **无**；任务结束后或 CI 里跑 |

这对「刷 star 的 demo」够用，对「监督员」叙事 **不够**——需要承认并规划升级。

### 4.2 目标形态（按易用性排序）

1. **Claude Code Hook（SessionEnd / PreToolUse）** — 任务结束自动体检；危险命令前警告（v0.2）
2. **本地 Web 仪表盘** — `ruledoctor watch` 监听 jsonl 增量，浏览器红绿灯（v0.3）
3. **MCP Server** — Cursor 侧边栏看规则状态（v0.4）
4. **VS Code/Cursor 插件** — 与编辑器规则文件联动（v0.5）

**一句话包装：** 「装完 hook，Agent 每次干完活自动交一份规则体检单；跑偏时复制纠偏 prompt。」

## 五、实时可视化（你想要的监督面板）

### 5.1 v0.1

- 终端 / HTML **静态报告**（事后）
- 无「当前任务」「进行中偏离度」

### 5.2 v1 监督面板应显示的字段（产品规格）

- 规则列表（来源文件、优先级占位）
- 每条：读到状态 / 遵守状态 / 严重程度
- 展开：证据（jsonl 行、文件行、命中片段）
- 会话：当前 session id、最近 tool call
- 任务偏离（可选）：与用户写的「本任务目标」对比 — **需额外输入或 probe，非 v0.1**

## 六、发现问题之后怎么办？（纠偏闭环）

### 6.1 v0.1

- 输出违规列表 + CI fail
- **没有**自动插入 prompt、没有拦截 tool

### 6.2 规划中的「下一步动作」

| 动作 | 实现思路 |
|------|----------|
| 可复制纠偏 prompt | `ruledoctor fix R6` 生成「你违反了…请…」 |
| 自动提醒 | Hook `SessionEnd` 把摘要贴回终端 |
| 阻止危险操作 | Hook `PreToolUse` + 与 `forbid-command` 联动 |
| 总结给 Agent | 输出一段适合粘贴回聊天的 markdown |
| 自动修代码 | **不做**（避免 scope 爆炸）；只给指令 |

## 七、定位：我们到底是哪一类产品？

建议 **对外只用一层主定位**，其它能力当阶段里程碑：

| 阶段 | 主定位 | 副标题 |
|------|--------|--------|
| **现在 v0.1** | **Agent 规则事后审计器** | 「会话结束后的规则体检」 |
| **v0.2** | **规则清单 + 证据链工具** | 「让你知道它在检查什么」 |
| **v1** | **Agent 规则监督员（Watchdog）** | 「实时红绿灯 + 一键纠偏」 |

**不是：** 通用 prompt 管理平台、也不是 Scale/Skills 编排器。  
**可以是：** 上述平台的「合规与到达率」插件。

## 九、建议的对外话术（统一口径）

- **问题：** 「规则写了，Agent 不听，还不报错。」
- **我们：** 「用会话日志当 ground truth，分清是没读到、还是没遵守。」
- **今天你能做：** 跑 `ruledoctor inventory` 看检查范围 → 配 `.ruledoctor.json` → 跑体检 → CI 卡点。
